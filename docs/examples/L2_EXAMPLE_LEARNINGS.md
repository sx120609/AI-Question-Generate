## 示例类型

特点：一个官方 PDF 附件，输出手机可打开 HTML，任务围绕条款解读、消费者截图举证、附件未覆盖处标注“办法未明确”。

第 14 行：刚毕业团队是否启动互联网创业。  
特点：企业开办流程、互联网信息服务备案/许可、CNNIC 报告三类资料；输出 Markdown 准备清单、事项判断表、3 个月行动路线，并在几个选项中给明确建议。

第 15 行：敏感肌面霜小红书/抖音种草测试与广告合规。

## L2 题目共性

题面像真实用户在交代工作，不像模板题。通常先给具体身份/处境，再给明确痛点，再给已有资料或约束，最后要求一个可执行判断。

题目必须有明确主决策，常见句式是“在 A/B/C/D 中给出一个明确建议”，不能只说“看情况”。这能防止模型输出泛泛分析。

L2 不一定都很长，但必须有真实业务流程感：读取资料、核验来源、拆解约束、对比方案、给出行动建议或可交付文件。

附件在题面里不用机械逐个长篇介绍，但字段里要说明附件来源、用途和证明边界。附件要能支撑题目决策，不能只给宏观资料去做具体个人/公司决策。

做题关键步骤一般是 6-12 步。下面是常见动作集合，不是每题照抄的固定顺序：
1. 先明确用户真正要解决的问题，防止跑题或扩大为年度全案。
2. 核验附件发布时间、发布主体、适用范围和现行有效性。
3. 分别读取附件，提取能支撑判断的关键事实、规则或数据。
4. 标出附件无法证明的信息，不让模型脑补。
5. 建立分类、对比、评分、情景或风险判断框架。
6. 生成指定产物，并做来源、格式、完整性和可用性自检。

打分 checklist 通常比关键步骤更细，常有 8-14 条，重点覆盖：是否完成核心任务、是否正确使用每个附件、是否给明确结论、是否避免编造、是否符合产物格式、是否保持真实工作语气。

## 产物写法

轻中型 L2 常见产物：
- Markdown 建议书 / 风险初筛建议 / 决策报告
- 分类表、对比表、行动路线、验证清单
- HTML 手机页或可打开的参考页

重型 L2 常见产物：
- Excel 分析文件，包含数据看板、评分模型、成本测算、财务模型、最终建议
- Markdown 管理层汇报简报
- 图表、三情景测算、公式动态计算和可追溯数据源

产物描述要具体到数量、格式、结构和验收点，例如“1200-2000 字 Markdown”“一张风险表达判断表”“前 3 个月行动路线”“五张核心工作表”。

## 生产时要学的语气

题目应使用自然、具体、带真实限制的用户语言，例如“预算 8 万元”“三个人能全职试 3 个月”“首年控制总投入不超过 300 万元”“还没走 IT 安全审批”。

不要把题目写成抽象命令，也不要堆很多不相关需求。一个题目应围绕一个主决策展开，次级任务都服务于这个主决策。

“不编造、来源未覆盖、任务不扩张”等通用约束由事实账本、事实锚点检查和验收规则承担。只有某个资料缺口会改变当前决定时，题面才用一句自然话交代具体缺口，避免把整批题写成免责声明集合。

## 对后续 L2 生产的执行口径

优先选择真实白领业务/工作流场景：运营、合规、投研、产品、市场、项目管理、区域进入、工具试点、客户交付等。

附件数量按二期规则优先控制在 4-8 个；示例中也存在 1-3 个附件的轻 L2，但我们生产时除非场景天然很轻，否则应尽量增加可支撑决策的附件组合。

每道 L2 至少要有：
- 一个具体场景和主决策
- 明确附件与来源
- 一个或多个可编辑/可打开产物
- 8 小时以上人类完成时间
- 8-15 步做题关键步骤
- 可执行 checklist
- 可回到来源的事实锚点，以及独立于写作者的事实审核

如果题目要求 Excel/HTML/PPT/Word 等文件，必须写清可打开、可编辑、公式/交互/图表/移动端等验收项。
