# L2 去 AI 风格管线

本文件定义 L2 题目生产中的固定去 AI 步骤。目标不是骗过检测器，而是让题面像真实需求方在飞书里交代工作。

## 参考原则

参考中文去 AI 项目时，不能只抄禁词表。L2 题面去 AI 要抓六件事：

1. 保真优先  
   不改事实、附件、时间、交付物和业务边界；没有资料支撑的内容不补。

2. 角色因果先于句式  
   先确认是谁在什么业务阶段发起请求、他知道什么、不知道什么、能决定什么以及结果拿给谁用，再让该角色说话。角色卡只保存当前任务所需的有限世界信息，不能用人物小传、性格标签和戏剧冲突制造“人味”。

3. 先判文体  
   L2 题面不是公众号文章、不是产品宣传，也不是模型回答。它应当像白领同事、老板、客户或项目组在交代一件具体工作。

4. 清结构壳  
   重点清理“不是...而是...”“真正重要的是...”“本质上...”“首先/其次/最后”这类模型常用骨架。能改成具体业务取舍，就不要用抽象判断。

5. 清语言腔但保留请求  
   去掉赋能、闭环、全链路、多维度、深度洞察、未来可期等空泛词；删掉假互动、客服式收尾、连续堆叠的礼貌祈使和鸡汤总结。“请”不是禁词，正式表人工直通稿中的“请结合、请只基于、完成后请检查”可以保留。B 列需要自然出现明确请求动作，让人直接看出要模型做什么、交付什么；任何请求示例都不是固定轮换表。

6. 正例用于校准，不用于仿写  
   保存真正自然的完整题面及其话语动作，不把样本只压缩成长度、附件数和步骤数。正例与机器反例主要供审核 agent 判断请求是否成立、角色因果是否可信；requester agent 不拿某条正例当语气模板。可以学习真实麻烦怎样产生请求，但禁止照搬开头、转场、标点节奏或句子。

## 角色视角前置阶段

去 AI 不能等整行 TSV 写完后再做表面改写。每条题先完成以下隐藏阶段：

1. 建立带稳定 `fact_id` 的事实账本，区分事实、未知事项、主决策和真实交付用途。
2. 现场选角 agent 生成隐藏角色卡与微型工作世界：`personaId`、组织环节、岗位职责和权限、当前阶段、已知/未知事实、收件人、结果用途、沟通环境和禁止越权知识。没有 `fact_id` 的老板催办、会议、截止时间、预算、冲突或事故不得进入设定。
3. requester agent 只扮演该角色，独立生成 3 个 B 列候选和 request sidecar；它不能看到整行 TSV，也不写 G/L/N/O。
4. 独立审核 agent 逐个检查三个 B 候选的知识越权、故事化添戏、角色互换、同作者遮罩、sidecar 原文和交付格式，只选择或判退，不拼接候选、不代写 B 列。
5. 字段编译 agent 只为获选 B 填写其余列，B 列逐字冻结；矛盾只能输出 blocked 并退回 requester/审核阶段，不能由编译器润色。
6. 获选记录按 `scene-card-bundle` 写入 `sources/scene_cards.json`，由 `runSceneCardGate` 对最终 TSV 复验角色卡、事实账本、sidecar、获选 B、事实边界和 M/B 映射，并在 `feishu/role_consistency_report.json` 输出 `kind=scene-card-gate-report`、`gateId=scene-card-role-consistency-v1` 的 `PASS/FAIL` 报告。报告绑定 candidate/bundle/fact-ledger/evaluation 哈希，并含逐行和遮罩整批相似审计；只有 `PASS` 才进入自然度、结构和 release gate，任一绑定哈希变化后旧报告失效。

request sidecar 使用运行时统一结构：`requestContract.requestSpan/action/outputs[{format,humanName,purpose}]`、`roleTrace.blockageSpan/motivationSpan/downstreamUseSpan`、`usedFactIds` 和 `deliberatelyOmitted`。`requestSpan` 与 `roleTrace` 中非空 span 必须在 B 列逐字且只出现一次；`action` 必须是 `requestSpan` 中逐字动作短语，`humanName` 必须是题面原词并逐字出现在 `requestSpan`；used 与 omitted 互不重叠并完整覆盖 `sceneCard.informationBoundary.knownFactIds`。sidecar 是证明材料，不能替正文补足请求。`沈礼`、`裴硬`仅是系统标注身份，每题请求者另用独立 `personaId`。

## 固定处理顺序

候选完成角色视角前置阶段后，必须进入以下诊断与返工管线：

1. 事实锁定  
   列出不可改字段：附件名、来源时间、唯一主决策、人类完成时间、产物格式和任务完成所需的真实动作。步骤必须保持 8-15 条，但不能为了保留旧数量而留下重复或无关动作。

2. 五字段按职责返工  
   不再只检查 `题目`，但也不要求五个字段为了显得不同而全部换骨架。B 列问题退回 requester agent，G/L/N/O 问题退回字段编译 agent；审核者不得直接改写 B。保持事实、附件和格式不变，优先修正真正重复的长句、任务概括、产物承接句和步骤正文。结构分类只作诊断；同为建议、清单或核对表不构成失败，只换“老板/客户”、行业名或日期则仍属于换皮。

3. 业务边界归位  
   先在事实账本中记录哪些信息没有来源。只有缺失信息会真实改变当前任务时，题面才自然交代一次；其余边界放进附件说明、待办字段或步骤，不再把整段免责声明当成题面内容。把“不要写成...”“不要编造...”“必须列出...”改成业务说法：
   - “结果要能拿去和产品/法务/老板沟通”
   - “公开资料无法确认的内容放进待确认栏”
   - “税务部分只给准备清单和会计师确认点”
   - “现场问题按业主、承包商、会计师分组”

4. 壳句清理  
   逐项检查；命中 B 列时退回 requester agent，命中其他字段时退回编译 agent：
   - 礼貌祈使壳：系统生成记录不使用“请”；需要明确委托时改用符合角色口吻的“帮我、给我、我想要、需要你、麻烦你”或自然工单表达
   - 二分壳：不是...而是...
   - 本质壳：真正关键的是、本质上、核心在于
   - 路标壳：下面我们来、接下来、总的来说、综上所述
   - 排比壳：不仅...而且...更...
   - 机械顺序壳：题面里的长串先...再...；详细步骤应放在 `做题关键步骤`
   - 假互动：你觉得呢、欢迎讨论、希望以上有帮助
   - 无请求规格句：Word需要……、Excel按……、本次任务需完成……

5. 生成形态检查  
   - 不使用统一 `questionTail` 给题面补尾巴；需要的信息直接写入合适位置，外部 QA 的每条建议也不得逐句追加到题尾。
   - 不为制造表面差异轮换“证据卡、资料块、核对问题”等附件样式。附件按资料本来的关系说明，标准的附件序号和来源标签允许一致。
   - 每题只保留一个主问题。敏感性、监测、后续研究等内容只有直接服务主决策时才保留。
   - 交付物位置由角色在当前沟通场景中的自然表达决定，不统一规定先后；但 B 列必须用一次清楚委托提出 M 列中的全部交付物，不能孤立成规格句。`产物内容` 再展开结构和验收点；只有 M 列 `产物格式` 写 `docx, xlsx` 等原始扩展名。
   - B 列题面可以连写，也可以按业务意群自然分段；段间只能有一个换行符，严禁连续换行造成空白行，不设置固定段数。`做题关键步骤` 仍按编号逐行换行。

6. 局部粗糙度检查  
   真人题面可以有一点工作口吻和不均匀节奏，但不能靠随意口语遮盖逻辑。允许事实自然形成长短句和局部停顿，不给每题统一添加“我这边”“老板催了”、会议或截止时间来伪造真人感。开头仍在描述同一触发事件时优先用逗号承接；冒号和分号看位置与语义，不设机械个数。顿号枚举过密、前 80 字频繁出现冒号分号、短状态句反复立刻句号，都进入整批复核。

7. 批量与历史正文检查  
   对题面及多字段正文检查共享长句、词法重复和换名词式改写，同时把开头、信息顺序、产物和步骤分类作为复核信号。共享 24 个连续归一化字符或正文高度重复属于硬失败；结构分类相同只进入复核，不强迫改成另一种产物拓扑。固定扩展名、附件编号和 `来源：` 不计入重复惩罚。

   另外必须做两项语义检查：角色互换测试中，把请求者换成职责明显不同的岗位，若原话无需改变仍完全成立，角色只是贴纸；同作者遮罩测试中，遮去公司、产品、行业、地点和附件专名后，若多题仍共享完整开头、转场、请求骨架、标点节奏和收尾，视为同一作者模板。两者失败都不能只换同义词。

8. 长度与事实密度检查  
   长度分布来自真人正例，只作诊断，不作为补字目标。题面偏短时先检查是否缺真实业务事实；事实不足就补来源、缩小任务或换题，禁止用缺件清单、免责声明、Excel字段和虚构会议补长度。批次若大量贴在同一窄区间，即使逐条达标也视为模板信号。题面不强制同时出现时间、缺口或会议，但必须有明确请求并说出全部交付物；M/N列不能替代这件事。

9. 本地 lint、事实门禁与整批自然度门禁  
   先验证数字、日期、引号内说法和截止时间都能回到来源，再校验角色卡、事实账本、request sidecar 和获选 B 的哈希及原文 span，最后运行 `lint_ai_style.mjs`、结构门禁和整批自然度门禁。自然度除原有指标外，还检查明确请求、M/B 交付物映射、请求框架集中度、短开头、第一枚标点、逗号/句号比、顿号密度、前段冒号分号和超长逗号句。`FAIL` 必须返工；`REVIEW` 未经独立签核也不得生成提交 receipt。

10. 审核反馈回灌  
   独立审核原因先归纳为根因，再决定修改题面、附件、产物、步骤还是规则。外部逐行 QA 负责可执行性、附件和格式，不等同于整批自然度审核；其示例句只是诊断线索，禁止照抄或逐句补回题面。同一反馈不能不断形成新的尾段和免责声明。

## 放行标准

- 题面开头不再像“为某某生成某某”的任务摘要。
- 系统生成叙述字段不使用“请”，同时保留明确的发起人、请求动作和交付用途，不能退化成无主语规格单。
- B 列有一次自然、明确的委托，能只看题面就抽取主任务和全部交付文件；不能只有 Word/Excel 的无主语规格。
- 题面没有“不要/必须”式提示词约束。
- 每个句群都有真实工作信息，不靠口号和总结句撑篇幅；同批题没有明显同款语义骨架。题面使用不超过 8 个自然段，不写项目符号规格单。
- 资料边界表达自然，读起来像“待确认事项”，不是像模型安全免责声明。
- 产物与用途在角色消息中形成清楚因果，并通过符合角色口吻的明确委托落到 Word、Excel、PPT 等格式；二者不强制固定先后。“帮我/给我/我想要/需要你/麻烦你”和自然工单表达都只是示例，同批不统一使用一个请求框架。
- M 列只写扩展名短标签；题面和 `产物内容` 不出现生硬的 `docx`、`xlsx` 原始标签。
- 题面长度由事实量决定，整批不贴着同一字符下限；没有靠附件复述、安全免责声明、虚构日期或字段枚举撑长度。
- 候选稿新增的数字、日期、引号内说法和截止时间均能回到来源或显式审核过的工作标签。
- 获选 B 逐字来自 requester agent；角色卡维持有限视角，没有故事化添戏，request sidecar 的 span、交付物和 `fact_id` 映射全部成立。
- 角色互换测试、知识越权检查和同作者遮罩测试均通过；沈礼/裴硬只作为标注身份出现。
- 整批自然度报告 PASS；如出现 REVIEW，存在独立审核人、候选哈希和逐项结论，不允许脚本自动接受。
- 同批与历史没有共享长句、正文高度重复或换名词式改写，并已生成有效 PASS receipt。
