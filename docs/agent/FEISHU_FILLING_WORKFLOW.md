# 飞书正式填写流程

这份说明记录 L2 题目正式填入飞书表格时的操作口径。核心原则：文字字段可以自动填写，附件字段必须上传真实文件对象，外部 AI 质检只作为提示。

## 填写范围

- 只填写「质量检验 checklist」之前的列。
- `UID` 暂空。
- 如表格有 `标注专家姓名`，从 `config/generated_identities.json` 选定当前 run 身份；目前为 `沈礼`、`裴硬`，姓名必须与 UID 前缀一致。
- `任务类型` 必须用飞书选项值 `L2 流程型`。
- `人类完成时间` 写 `10h`、`12h`，不写 `10小时`。
- `附件格式` 和 `产物格式` 都只写真实小写扩展名短标签。产物格式例如 `docx, xlsx`、`pptx`、`html`，不加说明性中文或括号。附件格式必须和相关附件真实扩展名一致。

## 并发填写边界

多线程生产时，不能直接“看到哪里空就写哪里”。正式写飞书前必须：

1. 创建本次 run 目录，记录 run id。
2. 通过 `build/automation/run_context.mjs` 获取 sheet 锁。
3. 根据当前表格底部空行预留连续行号。
4. 先写最左侧 A 列 UID 占位，UID 使用 `标注人_日期_编号` 风格，例如 `沈礼_7.9_01`。占位成功并 verify 后，才继续写其他列。
5. 只写本 run 预留的行；不得修改其他 run 的行。
6. 质检也只跑本 run 预留行。

如果没有拿到锁或没有 reservation，不允许提交飞书。

## 附件列

`相关附件` 是文件上传列，不是说明文字列。

正确做法：

1. 资料先下载到 `outputs/attachments/<题目名>/`。
2. 文件按内容重命名，保留真实扩展名，文件名必须按飞书样例加附件序号前缀，并尽量用中文主体名称，只保留必要英文品牌名/机构缩写，例如 `附件一_美国能源部替代燃料站数据下载说明.html`、`附件二_公共EV充电站选址检查清单.pdf`。
3. 本地 TSV 的 `相关附件` 字段只作为追踪清单，写真实文件名。
4. 正式填飞书时，选中对应 J 列单元格，走 `插入 -> 附件` 上传这些文件。
5. 上传后复核单元格显示蓝色附件链接，且文件名、`附件格式`、`附件内容` 三者一致。
6. `附件内容` 必须按附件逐项写清资料名、用途和来源链接。推荐写法：
   `附件一：《资料名》，用于...。`
   `来源：https://...`
   `附件二：《资料名》，用于...。`
   `来源：https://...`
   最后一行补 `边界：...`。
7. 附件优先保留真实原始资料：官方中文页面、原始 PDF、原始 CSV/XLSX/DOCX/API 文档等。不要为了通过检测器，把英文原始资料改造成“目标式中文摘要附件”来替代真实来源。
8. 如果原始资料是英文，在 `附件内容` 列逐项补完整中文摘要、用途和边界；必要时可以另附中文译文/摘要，但必须标明它是辅助阅读材料，不能用它伪装成原始资料来源。

禁止做法：

- 不把“中文摘要”“资料说明”“本地路径”写进 J 列冒充附件。
- 不只填 URL，也不省略 URL；每个附件要有中文用途/摘要和对应来源链接。
- 不把所有资料强行另存为 HTML 或 PDF；能拿到 DOCX、XLSX、CSV、JSON、图片、ZIP 等原文件就保留原格式。

## 浏览器自动化口径

文字列优先使用 `docs/agent/FEISHU_API_AUTOMATION.md` 中的 OpenAPI/官方 `lark-cli` 路径填写，不再用浏览器坐标批量粘贴。正式生产前先按 `docs/agent/FEISHU_USER_AUTH_AND_CLI.md` 检查用户身份授权；写表命令默认带 `--transport=lark-cli`。

如果直接 OpenAPI token 和官方 CLI 用户授权都不可用，浏览器自动化只允许作为临时兜底，并遵守两条硬规则：

- 先在 A 列写 UID 占位，避免别人抢行。
- 正文列一次只写本 run 已占位行；写前再次确认 A 列 UID 属于本 run。

附件列使用同文件里的 `uploadFeishuCellAttachments`：

```js
await uploadFeishuCellAttachments(tab, "J376", [
  "C:/.../附件一_Apple_App审核指南_官方中文页面.html",
  "C:/.../附件二_FTC_背书与推荐广告指南_2023.pdf"
]);
```

上传后使用 `verifyVisibleAttachmentNames` 做可见文件名复核。不要用 Windows 系统级粘贴文件，因为焦点可能落到 Codex 或其他窗口。

## 外部质检

外部 AI 质检不是最终裁判，以下情况可以本地复核后覆盖：

- HTML 或 PDF 附件来自英文官方来源，但 `附件内容` 已有中文摘要、用途和证据边界。
- 文件名里有 Apple、Google、IRS、API、CSV 等必要英文缩写，被误判为英文附件。
- `做题关键步骤` 在 TSV 中用 `\n` 暂存，粘贴飞书后已转换为单元格内真实换行，但质检仍解析成 1 条。
- 已上传真实文件对象、J 列展示名为中文主体、L 列已逐项写完整中文摘要和来源 URL，但外部质检仍提示“英文附件”。这种情况通常是质检端读不到文件对象正文、把英文官方原件、扩展名、来源 URL 或少量缩写算成英文，需记录本地覆盖依据。
- 不要为了消除外部误报而把真实附件替换成看起来专门服务质检的“中文要点附件”。人工审核优先看资料是否真实、可追溯、与题目匹配。

以下情况不能覆盖，必须返工：

- J 列没有上传真实附件文件。
- 附件打不开、格式不对、文件名和内容不对应。
- `附件内容` 没有中文要点，只堆英文链接或英文原文。
- `附件内容` 没有逐项说明资料名、用途、来源 URL 和边界。
- `附件格式` 与真实附件扩展名不一致，例如上传了 PDF 却只写 `html, csv`。
- L2 关键步骤少于 8 条，或步骤没有明确的核验、抽取、判断、产物生成和自检动作。
- 产物格式不是小写扩展名短标签，或题面没有用明确请求动作提出 M 列中的全部 Word、Excel、PPT、网页等交付物；产物内容还需展开使用对象和验收点。

## 人工审核修正

人工审核结果优先看左侧 `内部质检`、`原因`、`质检员`、`是否修改` 等列，不要把右侧 `LLM初步质检意见` 当成人工意见。

处理人工意见时：

1. 只修改原因列明确指出的问题字段。例如原因写“题面ai感，说教式约束句式（不要...）”，优先改 `题目`，不要为了通过机器质检去替换真实附件。
2. `原因` 和 `质检员` 保留原审核记录；修改完成后在 `是否修改` 列标记 `已修改`。
3. 同步修改本地生产脚本、风格检查器和文档，使下一次生成前就能拦住同类问题。
4. 对“不要写成/不要编造/必须问”这类题面提示词口吻，改成自然业务边界：哪些内容进待确认栏、哪些只给准备清单、结果给谁沟通、交付物如何使用。
5. 修改后重新生成填表 payload，跑本地 lint，再写回飞书并确认保存到云端。

## 填写后复核

每行至少复核：

- J 列显示真实附件链接，不是普通文本。
- K 列格式短标签与附件实际扩展名一致。
- L 列逐项说明资料名、中文内容摘要、来源 URL 和资料边界，不写用途话术，且附件序号与 J 列文件名前缀一致。
- O 列为 8-15 条逐行 ASCII 编号步骤。
- 飞书顶部显示“已经保存到云端”。

## API 优先填写路径

旧的浏览器坐标批量粘贴只允许作为临时兜底，不再作为正式填写路径。正式生产时按 `docs/agent/FEISHU_API_AUTOMATION.md` 执行：

1. 本地生成 `outputs/feishu_fill_plan_*.json`，并准备事实账本、`scene-card-bundle` 和角色一致性报告。
2. 从逐题状态机导出 `production_trace.json`，运行过程门禁生成 `l2-production-trace-gate-v2` 回执；候选稿、填表计划、正式字段和真实附件文件必须逐项一致。
3. 运行组合门禁生成 `combined-release-gate-v2` 回执；角色一致性必须直接 `PASS`，自然度和结构也必须授权。
4. 用 `build/automation/feishu_sheet_submit.mjs` 生成 dry-run，并同时传入 `--release-receipt` 与 `--process-receipt`。
5. 检查 `outputs/feishu_api_value_ranges_*.json`，确认 B:O 里除 J 列外均会通过 Sheets `values_batch_update` 写入。
6. 检查 `outputs/feishu_attachment_upload_queue_*.json`，确认 J 列附件是真实文件路径且无缺失。
7. 只有确认行号、sheet id、锁、dry-run 和两份回执后，才允许 `--apply`；底层 OpenAPI 会在锁内再次逐格核验 B/G/L/N/O，并复验附件哈希。
8. J 列仍上传真实附件对象，不把文件名、URL 或本地路径写进单元格冒充附件。
