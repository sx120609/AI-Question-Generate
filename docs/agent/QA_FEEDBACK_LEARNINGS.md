# QA Feedback Learnings

本文件记录外部质检暴露出的真实生成问题。每次质检反馈被本地确认有效后，都要把规则沉淀到这里，并同步更新 `L2_PROMPTS.md`、`L2_PRODUCER_AGENT.md` 或本地 lint。

## 2026-07-11 正式表人工直通稿校正生成规则

证据：
- 重新读取《叶老师对接群正式作业表——7.08》的 `7.08 L2作业表`。苏子瞻01、02、04、05、06在人工审核列为直接“通过”；07、08为“一次修改”后“修改通过”；03因普通报告采用 Markdown 产物未通过。
- 五条直通稿题面约 716-1394 个可见字符，使用自然段落，并会自然出现“请结合”“请只基于”“完成后请检查”。这证明“出现请”与“存在换行”不能作为 AI 风格硬失败。
- 直通稿的稳定共同点是具体岗位与真实对象、7-8 个互补附件、附件证据与内部待补事实分界、2-3 个可编辑产物、13-15 步证据推导，以及一个边界场景对全部产物做一致性回查。

规则校正：
- 题面硬下限改为 700，800-1400 为自然参考，1450 以上检查复述，1500 以上不放行。
- B 列允许按业务意群自然分段，不规定必须 2 段或 3 段；段间只用一个换行符，连续换行造成空白行属于提交失败，项目符号和编号规格单仍禁止。
- “请”不再作为硬失败。只有同一字段出现 4 次以上、每段固定以“请”开头或连续写成礼貌祈使规格单时才警告。
- 生产方法以证据链为中心：来源核验、事实抽取、证据比较、处理规则、产物落盘、业务场景回查和跨产物一致性自检。
- M 列仍使用本项目的机器可读短格式 `docx, xlsx`，不照搬直通稿中的 `DOCX、XLSX` 或 `word、excel` 表面写法。
- 本节覆盖下方2026-07-10关于“请”硬禁用和B列强制单段的旧规则；旧段落仅保留为历史复盘。

## 2026-07-10 一体化成题导致“同一个作者扮演所有行业”

问题：
- 只在 prompt 里要求“像真人”、补自然请求和调标点，仍然是在让同一个 L2 生产 agent 同时看 B/G/L/N/O 与 TSV 字段。模型会先按表格规格组织信息，再给题面套一层口语，人物有岗位名却没有有限视角，最终仍像同一个合规材料编排员。
- 把“帮我、你给我、我想要、需要你、麻烦你”写成固定识别列表，只能保证有请求动作，继续使用会形成新的五句式模板；request regex 不能证明是谁在什么处境下为什么这样说。
- “故事编写者”思路能增强角色感，但若直接要求编故事，会自动补老板催办、会议、截止时间、预算、部门冲突和人物关系，反而破坏事实保护。

修复：
- 一次性成题拆为：事实账本 -> 现场选角 agent -> requester agent 只写 3 个 B 候选 -> 逐候选出戏审核和同作者比较 -> 选择一个 -> 字段编译 agent 冻结 B 写其余列 -> scene-card gate 复验最终 TSV。
- 现场选角只生成隐藏角色卡和微型工作世界，记录独立 `personaId`、组织环节、岗位职责与权限、当前阶段、已知/未知事实、收件人、结果用途、沟通环境和禁止越权知识；不写年龄、爱好、性格小传或戏剧冲突。
- requester agent 不看整行 TSV，同一角色和事实独立生成 3 个 B 候选。每个候选附隐藏 request sidecar，统一使用 `requestContract`、`roleTrace`、`usedFactIds` 和 `deliberatelyOmitted`；其中每个非空 span 必须在正文逐字且只出现一次，`action` 必须是 `requestSpan` 原文动作短语，`humanName` 必须是题面原词并逐字出现在 `requestSpan`，sidecar 不能替正文补请求。
- 独立审核在编译前增加知识越权、故事化添戏、角色互换和同作者遮罩测试。换成职责明显不同的岗位仍原样成立，或遮去公司/产品/行业/地点/附件专名后仍保留同款开头、转场、请求和收尾，均为失败；审核员只能选择或判退，不能拼接候选。
- 字段编译 agent 不得改动获选 B 的一个字或标点。发现 B 与事实、M 列、sidecar 冲突时输出 blocked 并退回 requester/审核阶段，不把规格说明反向塞进题面；scene-card gate 随后对最终 TSV 再验证一次。

生成期保证：
- 获选记录写入 `scene-card-bundle`，角色卡、事实账本、获选 B、request sidecar 和字段编译结果做哈希绑定；`runSceneCardGate` 生成 `scene-card-role-consistency-v1` 的 `PASS/FAIL` 报告，绑定 candidate/bundle/fact-ledger/evaluation 哈希并包含逐行、遮罩整批相似审计。任一绑定内容变化使旧报告、审核和 release receipt 失效。
- `requestContract.requestSpan` 与 `roleTrace` 非空 span 必须是 B 列逐字子串，`action` 必须逐字出现在 `requestSpan`，`requestContract.outputs` 必须与 M 列逐项对应且 `humanName` 使用题面原词并逐字出现在 `requestSpan`；`usedFactIds` 与 `deliberatelyOmitted` 必须互斥并完整覆盖 `sceneCard.informationBoundary.knownFactIds`。语义 sidecar 与语言门禁共同工作，不再只依赖固定五种请求开头。
- 组合回执升级为 `combined-release-gate-v2`，强制绑定角色一致性 `PASS`。提交器在拿到表格锁后再次验证，底层 OpenAPI 和浏览器物理边界还会把 B/G/L/N/O 的 address、field、value 与回执中的 fill plan 逐格比对；因此不能绕过主流程直接改正文，M 列格式维护仍可独立执行。
- `沈礼`、`裴硬`只代表系统生成记录的标注身份和 UID 归属，禁止当作题目主人公；每条题使用独立 `personaId`，返修继续由同一请求者视角完成。

下次预防：
- 角色卡只解决“谁在说”，事实账本决定“能说什么”，requester agent 决定“怎么说”，字段编译只负责结构化；任何阶段不得越权代替另一阶段润色。
- 角色设定只保留会改变选词、关注点和信息顺序的工作信息。人物小传、戏剧性和无来源情节不属于去 AI 手段。
- 多候选只能在同一事实内改变真实关注重点和自然措辞，不预分配句式、标点、开头类型或产物拓扑。

## 2026-07-10 v3 来源驱动写作与组合发布门禁

复盘：
- 严格排除“❌不通过”后，其他标注人的真实通过样本共 84 条；其中无礼貌祈使“请”的 30 条题面最短 646 字，P25 为 715，中位数 916，P75 为 1129，最大 1588。此前基于生成记录得出的较短长度口径不适合作为最终标准。
- 第一版结构护照为了制造差异，硬分配长度带、信息顺序、流程和产物拓扑，反而产生“会议简报 + 情景模型”“谈判简报 + 条款矩阵”“诊断报告 + 监测看板”等不合场景产物，外部 QA 多次判定为过度正式、术语堆砌和不自然。
- 逐条照抄 QA 建议、用统一 `questionTail` 补尾、轮换证据卡/资料块/核对问题，会让题面随复检次数不断变长，并把同一模板隐藏在表面样式下。

v3 修复：
- 题面硬下限 560，650-1200 为非强制自然参考；超过 1250 警示复述和堆砌，超过 1500 硬失败。去掉附件名和交付格式套话后，核心场景至少 240，不再轮换长度带。
- 题面围绕一个主问题，至少自然覆盖角色或使用者、事实或证据、主决策和明确交付请求；时间、会议与缺口只有来源明确或业务表达需要时才出现。附件细节、验收和边界放到最合适的字段。
- 不再分配结构护照，开头、信息顺序、流程和产物关系全部由原始事实决定。结构分类接近本身不判失败，但会进入阻断式 `REVIEW`；只有独立审核人的哈希绑定 `APPROVE` 才能继续，脚本不得自动接受。
- 共享连续长句、词法正文高度重复和只换行业名仍为硬失败；步骤动作相同只有在业务正文也高度相似时才失败。
- 外部 QA 先归纳事实、范围、自然度、格式或步骤根因，再重写对应字段并删除旧句。示例句不得照抄，反馈不得逐句追加到题尾。
- M 列只写 `docx, xlsx` 等扩展名；题面和产物内容使用 Word、Excel、PPT、网页等人类名称，不出现 raw 扩展名。

下次预防：
- 每轮 QA 修订先做“删除什么、替换什么、是否改变主问题”的检查，禁止只增不减。
- 不使用统一 `questionTail`，不轮换附件装饰样式，不为通过分类器生造阶段门、情景模型或固定产物拓扑。
- 8-15 条步骤保持真实完成顺序，合并重复动作，末尾保留必要的交付前核对。

## 2026-07-10 礼貌祈使句仍暴露生成口吻（历史规则，已废止）

问题：
- 沈礼、裴硬的重写稿虽然补足了场景、证据和交付用途，题面中仍反复出现“请 + 动词”式礼貌祈使句，包括“请把、请按、请给、请完成、请提交、请分析、请整理”。
- 这类句子和既有外部质检中的“像任务指令”“过于正式和结构化”“不是自然业务对话”属于同一问题。只删除“请”字会留下命令壳，不能算修复。

当时的处理（已被后续人工审核样本推翻）：
- 曾把礼貌祈使句整体设为硬失败；这会让题目失去明确请求动作，产生“Word 需要……、Excel 按……”式规格说明。
- 2026-07-11 复盘 7.08 作业表中苏子瞻人工通过样本后，现行规则已改为允许角色语境自然的“请结合、请只基于、完成后请检查”等表达，只对同一字段连续堆叠四次及以上的礼貌祈使发出警告。
- 现行生成期与提交期以“明确委托 + 证据边界 + 交付用途 + 自然段落”为准，不再把单个“请”字或礼貌请求句当成人机味证据。

## 2026-07-10 去祈使过度导致需求动作消失

问题：
- 上一版把“避免请式礼貌壳”错误执行成“尽量不用请求句”，又把“我需要一版”“最后给我”当成普遍风险，生成器于是改写成“Word需要……、Excel按……”的无发起人规格说明。
- 当前系统记录中明确的“帮我/给我/我想要/需要你/麻烦你”请求覆盖率一度为 0%，模型只看 B 列很难判断到底要做什么。
- 当前题面的逗号/句号比约 1.34，真人正例约 2.38；短于或等于 30 字的首句占 6/22，而正例为 0/14。冒号分号的主要问题是出现得太早、几乎条条都有；顿号密度也明显更高。
- 旧基准又把 76 条其他标注人的真实通过记录中 62 条仅因出现“请”排除，丢失了大量真人句法和标点样本。

修复：
- 继续避免生成稿里的“请 + 动词”，但 B 列强制出现一次“请求方或收件人 + 动作 + 具体交付物”；允许“帮我、你给我、我想要、需要你、麻烦你”。
- M 列每种格式必须在 B 列以 Word、Excel、PPT、网页等人类名称出现，请求动作和交付物需位于同句或相邻句。
- `naturalness_benchmark_v2.json` 保留其他标注人的真实通过记录，不再因单个“请”丢弃样本；生成侧的禁用词规则与参考样本选择解耦。
- 句子只按 `。！？!?` 划分，分号不再伪造额外完整句；新增请求框架集中度、短开头、第一枚标点、逗号/句号比、顿号密度、前 80 字冒号分号和超长逗号句检查。
- 单段合并不再插入段落间空格，B 列连续落盘；O 列步骤仍保留逐行换行。

下次预防：
- 缺请求时重写整个请求句群，不能机械在原文前补“帮我”。
- 标点按意群重写，不全局把句号替换成逗号，也不为通过比例写成超过 240 字的一口气长句。
- 同批相同请求框架超过策略阈值进入阻断式复核，避免把所有题统一改成“帮我生成”。

## 2026-07-10 题目过短与跨批结构模板化

问题：
- 对沈礼、裴硬重写前的 22 条旧快照审计后，题面可见字符最短 189、中位数 215、最大 445，当时 22/22 都是单段；这些是问题定位数据，不代表当前已提交批次。
- 22/22 的产物内容都是 Word 在前、Excel 在后；14/22 恰好 11 步；步骤动作序列两两 LCS 中位相似度约 0.818，存在 5 组完全同构。
- 普通题面字符 3-gram 最大相似度很低，说明关键词去重会漏掉真正重复的叙事顺序、产物拓扑和步骤任务图。
- 最新三条在旧 `lint_ai_style.mjs` 下仍可直接 PASS，证明继续追加提示词或局部正则不足以阻断。

第一次修复及后续校正：
- 第一版曾引入长度带和结构护照硬匹配；该方案造成过度结构化，旧 v2 的过渡阈值也已被上方 v3 规则废止。
- v3 使用 560 硬下限、650-1200 自然参考、1250 警示和 1500 硬上限，不再按批次分配长度；核心场景至少 240。
- 跨批检查以共享长句、词法正文和换名词式改写为硬门禁；结构分类相近进入阻断式 `REVIEW`。固定扩展名、附件编号和来源标签不计入相似惩罚。
- 飞书正文提交使用 composite release receipt 哈希联锁，同时绑定自然度与结构门禁、候选稿和 fill plan。任一未签核 `REVIEW`、裸结构回执或门禁后叙述字段变化都会阻断提交；成功写入后再把登记状态标为 submitted。
- 同 run 的不同选题恢复相互查重；固定产物格式不再抬高主题相似度。

下次预防：
- 每个名额先生成多个低成本主问题候选，排除相同主题和换皮任务后再研究附件、写完整题面。
- 正文碰撞时重写真正重复的 B/G/L/N/O 内容，不接受只换行业名、角色名、同义词或步骤数量，也不要求五个字段为差异而全部换拓扑。
- 旧记录作为 legacy 历史基线参与比较，但不在未授权情况下自动改写远端。
