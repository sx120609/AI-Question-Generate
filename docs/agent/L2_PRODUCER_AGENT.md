# L2 Producer Agent Specification

## 目标

专门生产可填写到飞书表格中的 L2 流程型题目。当前只填写「质量检验 checklist」之前的所有列，`UID` 暂空。

Agent 的产出不是最终答案，而是用于测试大模型的题目数据。每条题目必须像真实白领工作任务：有明确业务处境、材料来源、约束条件、可交付产物和可验收边界。

成题采用角色视角流水线，不允许一个 agent 同时看着 B/G/L/N/O 和 TSV 模板写完整记录。事实账本确定真值，现场选角 agent 创建隐藏请求者，requester agent 只写 3 个 B 列候选和 request sidecar，独立 agent 逐候选出戏审核并选择一个，字段编译 agent 再冻结 B 写其余列，最后由 scene-card gate 复验完整 TSV。

## 输出列

按飞书表格顺序输出以下字段：

| 字段 | 填写要求 |
| --- | --- |
| UID | 暂空 |
| 题目 | 只接收 requester agent 的获选原始消息，后续 agent 不得润色；正文可连写或按业务意群自然分段，不规定必须 2 段或 3 段。段间只允许一个换行符，连续换行形成空白行直接失败，也不得写成项目符号或编号规格单。围绕一个主问题写清业务处境、关键事实或证据、限制和结果用途，并出现一次“请求方/收件人 + 动作 + 具体交付物”的自然委托。“请”不是禁词，允许符合岗位口吻的“请结合、请只基于、完成后请检查”，但不能连续堆叠成模板；“帮我、你给我、我想要、需要你、麻烦你”也只是示例。硬下限为 700 个可见字符，800-1400 是非强制自然参考区间；超过 1450 检查复述和堆砌，超过 1500 不放行。 |
| 任务类型 | 固定填飞书选项值 `L2 流程型`。 |
| 一级目录 | 只能使用二期规则中的一级类目。 |
| 二级目录 | 根据一级类目和真实业务场景填写，可在规则给出的方向上细化。 |
| 三级目录 | 更具体的业务场景，例如「AI 会议纪要工具合规试点初筛」。 |
| 任务概括 | 1 句话概括核心任务和主决策。 |
| 相关附件 | 本地 TSV 中列出已下载、已重命名的真实附件文件名，必须带扩展名；正式填飞书时上传为文件对象，不用文字占位，不堆 URL 或英文长标题。 |
| 标注专家工作年限 | 一般 3-5 年；专业门槛高的题可 5-8 年。 |
| 人类完成时间 | L2 不低于 `8h`，通常 `8h-16h`；字段里只能写 `10h`、`12h` 这种短格式，不写 `10小时`。 |
| 附件格式 | 只写真实格式短标签，例如 `pdf, html`、`html, csv, pdf`、`docx, xlsx`。能下载原文件时必须保留原格式，不要把所有资料都转成 HTML 或 PDF，也不要在本列写中文说明。 |
| 附件内容 | 逐项说明来源、发布时间/适用时间、中文摘要或中文资料口径、能证明什么、不能证明什么。英文来源必须补中文摘要/译文，不在本列堆 URL 或英文原文。 |
| 产物格式 | M 列只写小写扩展名短标签，例如 `docx, xlsx`、`pptx`、`html`。多个格式用英文逗号加空格分隔；不要写 `Word, Excel`，也不要写 `Word文档（docx）` 这类说明性标签。需要邮件正文时放到产物内容里，不把 `Email` 当格式。Markdown 仅用于 README、API 文档等真实技术交付。 |
| 产物内容 | 明确真实使用对象、必要结构和验收点，自然写 Word、Excel、PPT、网页等人类可读名称，不重复 `docx`、`xlsx` 原始扩展名，也不统一套“最终产物为两个可编辑文件”的固定句式。 |
| 做题关键步骤 | 8-15 步，必须使用外部质检可识别的 ASCII 编号并按行分隔。每一步都对应本题真实动作和依赖，不为凑数拆分同一动作，也不为躲避结构相似度生造阶段门、情景分叉或循环复测；核对放在真实发生的位置，不统一放到末步。 |

补充填写：系统生成身份以 `config/generated_identities.json` 为准，目前为 `沈礼`、`裴硬`。每个 run 选定一个身份后，`标注专家姓名` 与 UID 前缀必须一致；批量维护默认同时覆盖这两个身份，不得误改其他人员记录。二者只表示谁名下的系统生成记录，不能充当题中请求者。每题主人公必须另建独立 `personaId`，并绑定隐藏角色卡。

## 一级目录

优先使用以下二期一级类目：

- 互联网与平台业务
- 科技软件与 AI 工作流
- 游戏与互动内容
- 品牌市场与电商零售
- 投资战略、专业服务与企业经营
- 金融服务与财富投研
- 教育科研与生命科学
- 法律、政务与公共服务
- 房地产与大宗资产

## 推荐工作流

如果用户要求“全自动生产、提交飞书、执行质检并根据反馈调整”，必须先阅读并执行 `docs/agent/AUTO_PRODUCTION_WORKFLOW.md`。该流程定义了 run 目录、飞书行号锁、选题去重注册表、三轮质检和 bug 判定口径。多线程生产时不得跳过该文件。

如果外部质检反馈被本地确认是有效问题，不只修改当前飞书行；还要同步更新生成规则或 lint，并把“问题 -> 修复 -> 下次预防”写入 `docs/agent/QA_FEEDBACK_LEARNINGS.md`。外部 QA 的建议句不能逐句复制或追加到题尾，先归纳根因，再重写受影响的字段并删除旧的重复内容。

0. 生成主问题候选  
   每个名额先生成至少 2 个只含来源可证角色、关键事实和唯一主决策的低成本候选，先排除事实越界、与同批或历史相同的主题与换皮任务，再进入附件研究。v3 只预留 `sourceDriven` 名额，不再分配开头、信息顺序、决策、证据、流程或产物分类。

1. 选定业务方向  
   先确定一级目录、具体角色、业务处境和一个主决策。不要一题里塞多个互不服务的任务。

   多线程场景下，候选题进入资料搜集前必须先写入 `build/automation/topic_registry.mjs` 管理的选题注册表。若与已有 active/submitted/accepted 题目相似度过高，必须换题；不能只换地点、公司名或产品名做换皮题。

2. 设计 L2 真实工作量  
   明确为什么这是 L2：4-8 个材料、多约束、8-15 个真实步骤、需要人类把关，最终产物可编辑可验收，但不涉及 L3 级完整工程。产物由使用场景决定；普通协调会、物业审批、幼儿园试点或产品评审优先使用要点、建议、行动清单和核对表，不为“结构差异”硬套谈判简报、情景模型、阶段门矩阵或监测看板。

3. 网页资料搜集  
   使用网页版搜索 MCP 作为首选资料入口，先找候选资料，再逐个打开原网页、PDF、DOCX、XLSX、CSV、JSON、PPT、图片、ZIP、数据接口或下载页核验。搜索结果页不能直接当附件依据。

4. 建立来源卡片  
   每个资料都记录标题、发布主体、发布日期或适用时间、URL、格式、用途、局限。只有通过核验的资料才能进入「相关附件」。

5. 附件下载与格式识别  
   找到附件后先下载到本地，文件名用资料内容命名，例如「政策名称_发布日期.pdf」或「平台规则_功能名称.xlsx」，不要保留随机 token、长串数字或无意义下载名。格式识别按 `Content-Disposition` 文件名、URL 扩展名、`Content-Type`、文件头签名四层核验；DOCX/XLSX/PPTX 按 ZIP/Office Open XML 签名识别，PDF 按 `%PDF` 文件头识别，图片、CSV、JSON、TXT、ZIP 也要保留原始格式。动态网页只有在无法直接取得文档文件时，才保存渲染后的 HTML 或截图。

   如果原始资料是英文，必须额外准备中文摘要或中文译文口径。飞书「附件内容」列写中文摘要、用途和证据边界；原始 URL、英文长标题和英文正文保留在内部来源卡或附件文件中，不堆进飞书字段。

   「相关附件」列不能用“中文摘要”“资料说明”之类文字替代文件。内部 TSV 可保留真实文件名便于追踪；正式填飞书时必须把这些文件上传到单元格里，单元格应显示蓝色附件链接。

6. 附件质量闸门  
   找到候选资料后先判断附件质量，未达标不得继续成题。必要时回到资料搜集阶段补资料或换题。

7. 建立事实账本与现场选角  
   从核验资料中建立带 `fact_id` 的事实账本，再由现场选角 agent 生成隐藏角色卡和微型工作世界。角色卡只保留组织环节、岗位职责与权限、当前阶段、已知/未知事实、收件人、结果用途和沟通环境，不补人物小传、性格标签、老板催办、会议或冲突。

8. 生成 B 列候选与 request sidecar  
   requester agent 只扮演角色卡中的请求者，不看整行 TSV。使用同一角色和事实独立写 3 个 B 列候选，并为每个候选生成隐藏 sidecar：`requestContract.requestSpan/action/outputs[{format,humanName,purpose}]`、`roleTrace.blockageSpan/motivationSpan/downstreamUseSpan`、`usedFactIds` 和 `deliberatelyOmitted`。`requestSpan` 与 `roleTrace` 中非空 span 必须在题面逐字且只出现一次，`action` 必须是 `requestSpan` 中的逐字动作短语，`humanName` 必须使用题面原词并逐字出现在 `requestSpan`。used 与 omitted 互不重叠，合并后完整覆盖 `sceneCard.informationBoundary.knownFactIds`；sidecar 不能替正文补足请求。

9. 出戏审核与多候选择优  
   独立审核 agent 逐项检查三个 B 候选的事实越权、故事化添戏、角色卡泄露、request sidecar 原文映射、交付格式一致性、角色互换和同作者遮罩，并与同批声音信号比较。遮去公司、产品、行业、地点和附件专名后，仍与同批题共享整套开头、转场、请求和收尾骨架的候选失败；换成职责无关岗位仍完全成立的候选失败。只在硬检查通过的候选中选择一条，不拼接、不亲自改写。

10. 编译其余表格字段  
   字段编译 agent 只为获选候选输出 checklist 之前的其余字段，UID 留空。对 B 列逐字冻结，不得调整措辞、标点或补入产物规格；发现 B 与事实、M 列或 sidecar 冲突时输出 blocked 并退回 requester/审核阶段。题面不要机械逐个介绍附件，附件细节放在「附件内容」字段。

11. 内部质检  
   checklist 不写入飞书。获选记录先按 `scene-card-bundle` 契约写入 `sources/scene_cards.json`，其中绑定 `protocolId=situated-requester-v1`、事实账本路径与哈希、`recordUid`、scene card、request contract、role trace、used/omitted。再由 `runSceneCardGate` 对最终 TSV 复验，把 `kind=scene-card-gate-report`、`gateId=scene-card-role-consistency-v1` 的 `PASS/FAIL` 报告写到 `feishu/role_consistency_report.json`；报告绑定 candidate/bundle/fact-ledger/evaluation 哈希，并含逐行和遮罩整批相似审计。只有 `PASS` 才继续检查真实性、L2 难度、附件支撑、产物可验收、来源中的时间是否准确、无编造和无同质化。

12. 去 AI 风格与结构诊断  
   最终输出前必须按 `docs/agent/DE_AI_STYLE_PIPELINE.md` 做一次独立诊断：定位模板句、空泛形容词、整齐但不真实的并列结构、过度解释和 AI 式安全废话。诊断 agent 只选择、判退和说明证据，B 列返工只能交回 requester agent。

   不再参考单一“先背景、再附件、最后交付物”的已通过样例。五个叙述字段都要检查共享长句和同款正文，但不要求为了显得不同而轮换句群数量、附件样式或产物拓扑。题面可以按真实工作意群分段，但禁止统一段落数量、统一 `questionTail`、证据卡、资料块等拼装手法；标准附件序号和来源标签可以一致。

   `题目`、`任务概括`、`产物内容`、`做题关键步骤` 可以自然使用“请”，但同一字段出现 4 次以上或每段都以“请”起句时进入警告；参考正式表人审直通稿的真实请求，而不是把“请”当 AI 判定词。题面仍不得直接堆叠“不要写成...”“不要编造...”“必须列出/必须问”这类提示词式约束。
   交付物与业务疑问必须有清楚因果，但不强制固定先后位置；B 列必须直接提出 M 列中的全部交付物，用 Word、Excel、PPT、网页等人类名称表达。M/N 列负责结构化真值，不能代替 B 列告诉模型要做什么。

13. 最终放行校验  
   scene-card gate 通过后，再对飞书列、L2 难度、附件质量、产物真实性、去 AI 风格和同质化风险做最后一遍 pass/fail。组合发布门禁必须生成与事实账本、角色卡、request sidecar、真人基准、候选、自然度报告、结构报告和当前填充计划哈希绑定的 release receipt；任何一项失败、未签 REVIEW、receipt 缺失或门禁后内容变化，都不得提交飞书。

14. 导出  
   默认导出为 TSV，方便整行复制到飞书；也可同步输出 Markdown 便于人工审阅。

15. 飞书正式填写  
   文字列按飞书选项和字段格式填写；「相关附件」列只上传真实文件，不粘贴摘要文字或本地路径。上传后逐行复核文件名、格式标签和附件内容是否对应，确认飞书显示“已经保存到云端”。

## MCP 使用定位

前一条 Codex 线程里跑起来的 DevSpace 可以作为资料生产工作台：用 MCP 客户端发起搜索、下载附件、写来源卡、跑本地校验脚本、沉淀 TSV/内部审阅文件。它适合做「可追踪的资料流水线」，不适合直接当「资料质量裁判」。

使用边界：

- DevSpace 负责连接本地工作区和工具执行。
- 网页搜索 MCP 负责发现候选资料。
- 浏览器或下载请求负责打开原网页、PDF、DOCX、XLSX、CSV、JSON、PPT、图片、ZIP、表格和数据接口，并尽量保存原文件。
- 本 agent 的附件质量闸门负责判断资料是否能入库。

安全要求：

- allowed roots 必须只指向当前题目生产工作区，不要开放用户主目录或磁盘根目录。
- tunnel 地址和 owner password 只用于受控客户端，不写进公开文件。
- 能用本地文件/脚本完成的校验，不让外部网页工具处理敏感文件。

## 网页搜索策略

网页版搜索更适合当前任务，因为它能核验资料是否真实、当前可访问、是否直达附件。推荐采用「搜索扩展 + 浏览器核验」的混合流程：

1. 用搜索引擎扩展候选来源  
   每题先搜 8-15 个候选资料，优先官方、监管机构、公司公告/帮助中心、产品说明页、行业报告、公开数据表、标准/指南。

2. 打开原始页面核验  
   必须进入具体网页、PDF、Office 文件、下载页或数据页面，记录标题、发布方、日期、格式和链接。不要只引用搜索摘要。

3. 控制资料组合  
   单题优先 4-8 个附件；正式表人审直通稿通常由 7-8 个互补材料形成“主证据—规则—数据明细—背景”链条，资料确实需要时优先使用这种上半区密度，但不得按数量配额堆附件。尽量混合 PDF、HTML/网页、DOCX、XLSX/CSV、PPTX、JSON/API 文档、产品页、图片或 ZIP，避免全是新闻稿。

4. 下载与命名  
   附件能下载原文件就下载原文件，并按内容重命名；文件名要让审核人不打开文件也能大概知道内容。下载后记录本地路径、真实格式、文件大小和 SHA256；格式与扩展名不一致时先排查，不要直接入库。

5. 处理时效性  
   涉及政策、平台规则、公司数据、产品功能、价格、市场数据时，附件内容记录来源日期或适用时间；题面只保留完成判断所需且来源可证的时间锚点。

6. 标出证据边界  
   附件能证明的内容要具体；无法证明的内容要写明「附件未覆盖」或「未获取到可靠公开数据」，禁止让模型脑补。

## 附件质量闸门

候选资料必须先过附件质量闸门，再进入题目生产。单个附件按以下标准判断：

| 维度 | 通过标准 | 不通过示例 |
| --- | --- | --- |
| 真实性 | 能打开原始页面、PDF、Office 文件、表格、公告或数据页；标题、发布主体和 URL 一致。 | 只有搜索摘要、二手转载、打不开、需要未知权限。 |
| 权威性 | 优先官方、监管机构、公司公告、产品文档、行业协会、研究机构、公开数据平台。 | 自媒体泛泛文章、营销软文、来源不明搬运。 |
| 直达性 | 链接直达具体资料或下载页。 | 只给首页、栏目页、搜索页。 |
| 时效性 | 政策、产品、市场、价格、规则类资料有发布日期或适用时间。 | 无时间、明显过期但题目要求当前判断。 |
| 相关性 | 能直接支撑题目的主决策或关键约束。 | 宏观背景很多，但无法支撑具体业务判断。 |
| 可处理性 | 文档可下载/可阅读，表格字段清晰，页面主要内容可复制或截图；本地文件真实格式与扩展名一致。 | 扫描模糊、乱码、无法提取关键内容，或把 DOCX/XLSX 等原文件误存成网页壳。 |
| 证据边界 | 能写清它证明什么、不能证明什么。 | 资料用途不清，被硬塞进题目凑数量。 |

整题资料包通过标准：

- L2 优先 4-8 个附件。
- 至少 3 个附件直接支撑主决策，不只是背景资料。
- 至少 1 个附件提供结构化事实、规则、数据或产品约束。
- 资料组合不应全部来自同一类二手媒体。
- 若涉及具体公司/城市/产品/政策，资料包必须包含可落到该对象的具体信息。
- 英文来源可以使用，但必须配套中文摘要/译文；飞书字段中的「相关附件」和「附件内容」应以中文为主，只保留必要品牌名、文件名或缩写。
- 如果资料包不达标，必须先补资料、换场景或降级为 L1，不得继续生产 L2。

## 外部 AI 质检口径

外部 AI 质检是辅助提示，不是最终裁判。它可能把 HTML 页面中的英文语法、文件名里的英文缩写或单元格换行识别错。处理方式：

- 先把反馈归类为事实/附件、范围、自然度、格式或步骤问题，再修改对应字段。不得照抄 QA 的示例句，不得把每轮反馈追加成新的题尾段、`questionTail`、免责声明或产物字段。
- 同一问题复检时以根因重写替代逐句补丁；新增一句前先删除已失效或重复的旧句，避免题面随轮次不断膨胀。
- 如果本地已确认附件真实、可打开、字段里有中文摘要/中文资料口径，外部质检仍提示“英文附件”，可记录为本地覆盖。
- 如果外部质检提示关键步骤不足，先本地核对是否为 8-15 条逐行 `1. ...` 编号；若只是质检解析失败，可覆盖；若确实少于 8 条或没有拆开动作，必须返工。
- 不得覆盖的硬问题：附件未上传真实文件、附件打不开、资料与题目不匹配、附件内容没有中文要点、关键步骤没有明确动作、产物格式不符合飞书选项。
- 覆盖前要写清本地判断依据，例如“步骤 10 条，TSV 粘贴时已转换为单元格内真实换行”“英文官方来源已在附件内容列补中文摘要和边界”。

## 去 AI 风格审校

去 AI 风格不是为了规避检测器，而是为了让题目更像真实工作需求，避免批量生成痕迹。最终输出前按以下方式调风格：

- 先核对隐藏角色卡，而不是先改句子。题面只使用该角色有权知道的事实和会关心的问题；普通经办人不能像全知法务、审计或管理层一样给出完整结论。
- 把「请全面深入分析」「综合考虑多维因素」「形成可执行方案」这类空泛话替换成具体处境、限制和要交付的文件。
- 系统生成记录不使用“请”。这条限制不能把题面改成“Word需要……、Excel按……”的无主语规格说明，仍要保留明确发起人、请求动作和交付用途。
- 删除或改写无主语命令句，例如「报告你要写清楚」「表格你要能算」。改成有对象、有场景、有交付物的表达。
- 检查逻辑断层。遇到转折、因果或并列要求时，必须能说清前后关系；不要把读取字段、核对清单、计算指标混成一类动作。
- 补齐必要主语、对象和业务场景。题面可以口语化，但不能省到读者不知道谁在做、给谁看、为什么现在要做。
- 删除无信息量的俏皮话和比喻。真实口吻可以保留，正式任务里不能用玩笑替代判断依据。
- 慎用「兜底」「底稿」「风险转嫁」「赋能」「闭环」「全链路」等行业黑话；只有场景确实属于审计、投行、法务、大厂管理汇报等语境时才保留。
- 默认使用白领工作身份，不要写成学生汇报、论文投稿或实验报告；除非题目本身属于科研课题、课程项目或实验室横向项目。
- 减少整齐三段式、排比句和过度工整的连接词。
- 标点按意群安排：开头仍在讲同一触发事件时用逗号承接，不反复用短状态句立刻句号；冒号只引出原话或真正短清单，分号只处理复杂并列，顿号过密时改回自然短语。修复不能靠全局替换句号或强塞逗号。
- 题面保留真实用户的轻微不完美表达，但不能牺牲清晰度。
- 不用「作为一名」「在当今快速发展的时代」「赋能」「闭环」「全链路」等高 AI 味表达，除非行业语境确实需要。
- 题面内部句群服从内容，不固定采用三段式语义骨架，也不为制造差异机械轮换开头或发问形态；B 列可连写或自然分段，段间只留一个换行符，连续空行直接失败。
- 同一批题不得都使用“我在...负责...”“附件里放了...”“最后给我...”这类同款开头和转场。附件、交付物和资料缺口可以写在不同位置，只要读者能看清谁要做、为什么现在做、结果给谁用。
- 避免把题面写成摘要句或任务说明书，例如“为某某进行预审，形成清单”。这类话可以放在「任务概括」，题面要像同事发来的具体需求。
- 避免把产物要求写成无请求方的规格句，例如“Word需要包含……，Excel按……”“本次任务需完成……”。“我想要一份……”“我需要你帮我……”本身是正常请求，只有整批重复同一框架时才进入模板化复核。
- 做角色互换测试：若把请求者换成职责明显不同的人，原话不改仍完全成立，角色只是贴纸，退回现场选角或 requester agent。
- 做同作者遮罩测试：遮去公司、产品、行业、地点和附件专名后，与同批或历史题仍共享整套开头、转场、请求、标点节奏和收尾，按模板化失败处理，不能只换同义词。
- 检查 request sidecar 的 `requestContract`、`roleTrace`、`usedFactIds` 和 `deliberatelyOmitted`。`requestSpan` 与 `roleTrace` 非空 span 必须逐字存在于正文，`action` 必须逐字出现在 `requestSpan`，`outputs` 必须覆盖 M 列且 `humanName` 使用题面原词并出现在 `requestSpan`；used 与 omitted 必须互斥并完整覆盖 `sceneCard.informationBoundary.knownFactIds`。sidecar 只能证明正文已有内容，不能用隐藏字段替 B 列补请求。
- 少解释附件背景，把附件细节放到字段里；题面只交代任务为什么存在。
- 每题保留来源中真实存在的约束，如预算、截止时间、团队能力、审批边界、数据缺口或老板偏好；来源没有就不补。
- 流程型任务不能只用几句话糊过去。关键步骤必须写出资料核验、字段抽取、冲突处理、判断依据、产物整理和自检放行。
- 最后读一遍：如果像投喂模型的说明书，而不像同事交代任务，就返工。

## 产物真实性闸门

题目要求模型输出的产物，必须符合真实职场里人会要求的交付形式。内部文件可以用 Markdown 审阅，但题目中的「产物格式」和题面交付要求默认不得写 Markdown。

推荐写法：

- `Excel/飞书表格`：测算表、评分模型、风险清单、数据看板、库存/预算/ROI 模型。
- `Word/飞书文档`：客户交付报告、管理层简报、合规初筛意见、调研纪要、操作指引。
- `PPT`：上会材料、老板汇报、客户提案、方案比选页。
- `HTML 页面`：可打开的移动端参考页、交互式 dashboard、产品对比页、活动页原型。
- `JSON/YAML/配置表`：仅限开发者、工具配置、规则配置等真实技术场景。

如果真实交付里需要邮件正文或通知文案，把它写在「产物内容」里，例如「Word 附录包含一段供应商邮件正文」；不要把 `Email` 单独写进「产物格式」。

不推荐写法：

- `Markdown 报告`
- `md 文档`
- `用 Markdown 输出分析结果`
- `生成一份结构化 Markdown`

少数例外：如果真实场景是开源 README、API 文档、开发者手册、GitHub issue 模板、技术知识库沉淀，可以使用 Markdown，但题面必须说明为什么业务方确实会收 `.md` 文件。

## 内部质检门槛

生成后逐条检查：

- 是否有具体角色、业务背景和一个主决策。
- 题面是否有真实需求方锚点：谁在做、什么时候要、卡在哪里、结果给谁用。
- 隐藏角色卡是否只含完成任务所需的有限世界信息；B 列有没有超出 `requester.authorityBoundary` 或 `informationBoundary.knownFactIds`，有没有无来源的故事化添戏。
- 是否符合 L2：人类时间 >= 8 小时，关键步骤 8-15 步，通常 4-8 个附件。
- 附件是否真实、直达、可打开，且能支撑题目中的关键判断。
- 是否避免只用宏观资料解决具体公司/个人决策。
- 产物格式和产物内容是否足够具体、可编辑、可验收。
- 产物是否像真实用户会要的交付物；默认不得要求 Markdown。
- 题面是否自然，像真实需求方在交代工作。
- `题目` 是否有明确请求动作并逐一说出 M 列交付物；自然“请”可以保留，但同一字段不得堆叠 4 次以上或成为每段固定开头。
- 是否没有无主语命令句、逻辑断层、无意义俏皮话、身份违和和错位行业黑话。
- 来源中的时间锚点是否准确，新增数字、日期、引语和时间词是否都能回到来源；通用资料边界是否由事实账本和门禁承担。
- 是否与同批及历史题目没有共享长句、正文高度重复或换名词式改写；相同自然产物不单独判定同质化。
- 是否已经完成去 AI 风格审校，且没有明显模板化句式。
- 题目是否达到 700 个可见字符硬下限，通常自然落在 800-1400；超过 1450 是否检查过复述和堆砌，超过 1500 是否已返工。题面围绕唯一主问题覆盖角色/使用者、事实/证据、主决策和明确交付请求，不强塞没有来源的时间、会议或缺口。
- B 列题面是否按业务意群自然分段、段间无空白行且不含项目符号/编号规格单；O 列步骤是否仍保留真实逐行编号。
- 获选 B 是否逐字来自 requester agent，request sidecar 中的非空 span 是否都能在正文找到，`requestContract.outputs` 是否逐项覆盖 M 列。
- 是否通过角色互换测试和同作者遮罩测试；沈礼、裴硬是否只出现在标注身份字段，题中请求者是否使用独立 `personaId`。
- 是否已与同批和历史正文比较；共享长句、词法高度重复和只换行业名属于硬失败。结构分类相同进入阻断式 REVIEW，不为通过门禁生造产物或流程；独立签核前不得生成 release receipt。

## 最终放行校验

每条题输出前必须逐项通过以下校验。只要有一项失败，就回到对应阶段返工。

| 校验项 | 通过标准 | 失败后的动作 |
| --- | --- | --- |
| 飞书列完整性 | 只包含 checklist 前字段；任务类型为 `L2 流程型`；标注专家姓名必须来自系统生成身份注册表，且与 UID 前缀一致；字段顺序与模板一致。 | 修正字段或重新导出。 |
| L2 难度 | 人类完成时间 >= `8h`，且必须写成 `10h` 这类短格式；关键步骤 8-15 步；需要多资料、多约束、多判断；没有升级成 L3。 | 调整任务范围、步骤和产物。 |
| 表格格式 | K 列附件格式、M 列产物格式只写真实扩展名短标签；M 列例如 `docx, xlsx`。 | 把 `10小时`、`HTML 网页`、`Word文档（docx）` 等冗余值改成字段要求的短格式。 |
| 外部质检格式 | 做题关键步骤使用逐行 `1. 2. 3.` 可解析编号；相关附件和附件内容以中文摘要/中文资料为主；M 列只写扩展名且不写 `Email`；B 列用自然请求句明确提出 Word/Excel 等全部交付物。 | 改编号和换行、补中文摘要/译文、把邮件正文放入产物内容，并统一扩展名短标签。 |
| 主决策 | 题目围绕一个清楚的业务判断，不是一堆互不相关要求。 | 重写题面和任务概括。 |
| 附件质量 | 所有附件已过质量闸门；链接直达；来源、时间、格式、用途、边界清楚。 | 补资料、剔除资料或换题。 |
| 证据匹配 | 附件能支撑题面中的关键判断，不能只用宏观资料解决具体决策。 | 补具体资料或降低题目目标。 |
| 产物真实性 | 产物像真实人类工作交付物；默认不要求 Markdown；可打开、可编辑、可验收。 | 改成 Word/飞书文档、Excel/飞书表格、PPT、HTML 等真实格式；邮件正文放入产物内容。 |
| 题面人味 | 语言自然具体，有真实约束和明确委托，不像模板题、考试题、产物说明书或 prompt；允许符合角色口吻的“请”，但不得连续堆叠礼貌祈使，也无无主语命令句、逻辑断层、错位黑话和学生身份违和。段落、标点和请求框架不过度集中。 | 进入去 AI 风格审校返工。 |
| 角色一致性 | B 列逐字来自 requester agent；题面事实均在角色可知范围内，没有故事化添戏；request contract、role trace、used/omitted 与原文、M 列、事实账本一致；`scene-card-role-consistency-v1` 对最终 TSV 为 `PASS`，candidate/bundle/fact-ledger/evaluation 哈希及逐行、遮罩整批审计均通过；沈礼/裴硬仅为标注身份。 | 按根因退回现场选角、requester 或字段编译 agent，审核员不得直接改写 B 列。 |
| 时效和边界 | 最新规则、政策、产品、市场数据的来源时间可追溯；题面时间与具体缺口均来自事实源。 | 修正来源时间；把通用边界移入事实账本，只保留影响决定的具体缺口。 |
| 同质化 | 与同批及历史记录没有共享长句、正文高度重复或换名词式改写；角色互换和同作者遮罩测试均通过；相同自然产物不单独判失败。 | 从真实角色、事实和主问题重写，不靠轮换附件样式或产物拓扑。 |
| 题面长度 | 不少于 700，不超过 1500；800-1400 为自然参考，超过 1450 已检查复述。 | 删除重复附件说明、边界句和逐轮 QA 补丁；事实不足时缩小任务或换题，不能虚构情节补字。 |
| 可粘贴性 | TSV 单元格不破坏列结构；换行、列表、分号等不会导致飞书错列。 | 清理格式后重新导出。 |

## 输出格式

默认输出两份：

1. `outputs/l2_questions.tsv`  
   用于直接粘贴飞书表格。第一列 UID 留空。

2. `outputs/l2_questions_review.md`  
   用于内部人工审阅，每题包含来源卡片和内部质检结果，但 checklist 不作为飞书列输出。注意这是生产过程文件，不代表题目要求模型交付 Markdown。
