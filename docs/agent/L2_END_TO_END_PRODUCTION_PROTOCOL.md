# L2 端到端生产协议

本协议是新题生产的前置总门禁。固定顺序为：可仿题面抽样 → 结构拆解 → 新附件构建 → 新题生成 → 第一道质量质检 → 第二道语言质检 → 最终字段编译与提交。任何阶段都不得跳过。

## 强制输入

每次运行必须重新读取并记录哈希：

1. 飞书 Wiki《垂域高难度题目生产--二期要求》（文档节点 `F2ZqwOpzKiST4PkNtDOcHdgSnCc`）；仓库使用从该文档拉取并归一化保存的 `docs/rules/02-垂域高难度题目生产-二期要求.md` 作为生产输入，并记录来源链接与内容哈希。
2. `inputs/production/可仿题面.xlsx`。
3. `inputs/production/L2题面第一道质检提示词_增强覆盖版.txt`。
4. `inputs/production/第二道质检_增强覆盖版.txt`。

缺少任一可用输入时，创建新生产 run 直接失败。不得使用旧 QA Prompt、旧反馈表或经验记忆替代。

## 单题流程

1. 每题从可仿表独立抽取一行，只把题面和附件内容概括交给后续 agent。表中的内部质检、提交状态、任务等级和其他字段不进入模仿上下文。
2. 先记录业务场景、核心卡点、唯一主任务、附件配合方式、产物形成原因、可模仿结构和禁止复用内容。
3. 新附件不得复用样本附件。经独立语义校验的对象级、执行级和连续业务材料必须达到整组的80%，政策、法规和行业资料最多占20%，只承担边界或背景作用。`specific-business` 标签不能自证，材料必须同时具备明确对象、时间或事件、独有业务内容；政策页面、管理规定、指南和解读不能冒充具体材料。趋势题优先使用连续时间序列，否则说明不影响判断的原因。
4. 新题全部动作服务一个主决策、主流程或主交付。人物、数据、限制和概念必须进入判断链。
5. 第一道质检严格使用原始提示词。该提示词的最终 pass/fail 只由“附件支撑是否充分”和“关键信息是否缺失”决定。唯一主任务、对象决策、证据矩阵和 L2 推理链属于执行第一道质检前的结构审计留痕，不得偷偷改写质检提示词的判定口径。
6. 第一道不通过时先修逻辑或附件；连续两轮仍不通过，废弃该题并重新抽样。只有第一道通过后才允许进入第二道。
7. 第二道只改语言，不新增事实，不改主任务、附件边界和产物类型。全文禁用分号；单句最多一个顿号；逗号不得伪装列表；用于列举收束的“等”全文至少三处，“等待”“等级”等普通词义不计；括号只在确有解释价值时使用，不设数量要求；禁用提示词中列明的机械句式。
8. 第二道必须逐一说明每对相邻句和相邻段的因果、解释、递进、转折、条件、时间推进、对象延续或任务收束关系，不能抽样。题面应让外行顺着上下文听懂，无法从上下文理解的专业词必须改成日常说法或随文解释，最终保持娓娓道来的叙事推进。
9. 最终 B 列和 L 列允许自然单换行，不得有空白行。附件内容概括只写附件实际包含的信息，不写“用于支持”或“为……提供依据”。
10. 产物格式从真实使用环节里选择，不得默认套用 `docx, xlsx`。汇报演示可用 `pptx`，数据跟踪或测算可用 `xlsx`，在线查看或交互交付可用 `html`，签发、打印和归档版可用 `pdf`，长篇说明或正式文稿可用 `docx`。`csv`、`json`、`md` 等只在数据交换、系统接口或开发文档场景中使用。每个格式必须写明使用者、工作环节和选择理由，不能为了覆盖率凑文件。

## 留痕与门禁

`production_input_packet.json` 记录四个输入的路径、哈希、完整质检提示词、样本位置，以及本批每题唯一的题面/附件概括样本。`production_trace.json` 必须逐题记录结构拆解、新附件支撑、两道完整质检结果、每轮修复和最终 14 个正式字段。

`production_workflow_state.json` 是实际执行状态，不是事后补写的说明。每题只能按 `REFERENCE_SAMPLED → STRUCTURE_READY → ATTACHMENTS_READY → DRAFT_READY → FIRST_QA_PASS → SECOND_QA_PASS → COMPLETE` 推进。返修会回到相应节点；第一道连续两轮失败进入 `ABANDONED_RESAMPLE_REQUIRED`，该样本不能继续编译或提交。六段可执行提示词由 `production_pipeline_prompts.mjs` 生成，第一、第二道质检正文直接取自输入包，不使用另写的简化版。只有结构拆解节点能看到完整抽样题面，生成节点只接收结构卡，不再接收原题全文。

运行 `production_trace_gate.mjs` 后，只有 trace、候选 TSV 和 fill plan 的行数、UID、14个正式字段及附件名称完全一致，并且本 run 中每个真实附件的路径、大小和 SHA-256 均可复验，才生成 `production_trace_gate_receipt.json`。该门禁还会重新计算单句顿号数、有效“等”的数量和逗号伪装列表，并校验叙事承接审计是否覆盖所有相邻句与相邻段、外行可懂检查是否清零未解释术语。格式门禁对5题以上批次要求覆盖 `docx`、`xlsx`、`pptx`、`html` 和 `pdf`，对10题以上批次还限制单一组合不超过40%、单一格式不超过75%，并要求至少5种组合；格式在题面或产物内容中没有真实用途也会失败。正式提交入口同时校验该回执与现有 naturalness、scene-card、structure、release 回执；门禁后修改附件、题面或产物格式都会让回执失效。

## 批量规则

批量中的每题独立抽样、独立附件、独立质检。批内抽样默认不重复。每题完成后记录失败点和修复点；同类问题重复出现时先形成候选规则，等待用户确认后再写入相应提示词或项目指令。结构门禁继续负责跨题、跨批次的后半段结构、产物组合、角色和流程去重。
